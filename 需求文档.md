## 2. 产品概述

### 2.1 核心愿景

打造一款**轻量、好用、跨端同步**的个人便签应用：
- **第一阶段（MVP）**：核心能力对齐 WPS 便签，满足"快速记录 + 待办管理 + 多端同步"。
- **第二阶段（迭代）**：在便签的基础上叠加智能化能力（智能分类、智能提醒、AI 辅助整理等），打造专属个人效率工具。

### 2.2 产品定位

> "**比 WPS 便签更懂我，比 Notion 更轻量。**"

| 维度 | WPS 便签 | Notion / 印象笔记 | Smart Todo（本产品） |
| --- | --- | --- | --- |
| 定位 | 轻量便签 | 重型笔记/知识库 | 轻量便签 + 智能化 |
| 启动速度 | 快 | 慢 | 快 |
| 富文本 | 支持 | 强 | 支持（够用即可） |
| 待办 | 嵌入式 checklist | 数据库式 | 嵌入式 + 独立待办视图 |
| 智能特性 | 弱 | 中 | **强**（个性化迭代重点） |

### 2.3 用户画像与使用场景

- **角色**：开发者本人
- **场景**：
  1. **PC 上规划**：在浏览器中创建/整理便签，规划本周任务、记录技术学习笔记。
  2. **手机随手记**：通勤、会议、灵感闪现时，在 Android 上快速记录。
  3. **离线可用**：地铁、电梯等弱网环境下，仍能查看与新增便签。
  4. **跨端同步**：手机记录、电脑整理；电脑规划、手机提醒。
  5. **到期提醒**：便签内的待办到期时，在系统通知栏弹出提醒。

## 3. 功能需求

> 标注说明：**[MVP]** 表示首版必须；**[V2]** 表示二期；**[迭代]** 表示个性化探索期。

### 3.1 便签管理（核心实体）**[MVP]**

便签（Note）是产品的核心实体，所有内容（文字、待办、图片）都属于某条便签。

- **增删改查**：创建、编辑、删除、恢复便签。
- **便签属性**：
  - 标题（可选；为空时自动取正文首行）
  - 富文本正文（见 3.2）
  - 创建时间、最后修改时间
  - 所属分组（见 3.4）
  - 颜色标记（见 3.9）
  - 是否置顶
  - 是否在回收站
- **基本操作**：
  - **置顶 / 取消置顶**：置顶便签固定在列表顶部
  - **复制便签**
  - **分享便签**：导出为图片 / 纯文本 / Markdown **[V2]**
  - **批量操作**：批量删除、批量移动到分组 **[V2]**

### 3.2 富文本编辑器 **[MVP]**

参考 WPS 便签的编辑能力，做到"够用且顺手"。

- **基础格式**：加粗、斜体、下划线、删除线
- **段落级**：标题（H1/H2/H3）、引用、分割线
- **列表**：无序列表、有序列表、**待办清单（checkbox）**
- **字体**：字号（小/中/大）、字体颜色、背景高亮色
- **媒体**：插入图片（本地上传 / 粘贴）**[MVP]**；附件 **[V2]**
- **链接**：插入超链接、自动识别 URL
- **撤销/重做**：标准 `Ctrl+Z / Ctrl+Y`
- **快捷键**：常见 Markdown 快捷键（如 `# ` 转标题、`- [ ] ` 转待办）
- **自动保存**：编辑过程中防抖保存到本地，避免数据丢失

### 3.3 待办清单（嵌入便签）**[MVP]**

待办作为便签内的一种**内容块**存在（而不是独立的实体），更符合 WPS 便签形态。

- **基本能力**：
  - 在便签内插入 `- [ ]` 待办项
  - 点击复选框切换完成/未完成
  - 已完成项可视化区分（删除线 + 灰色）
  - 支持调整顺序（拖拽）**[V2]**
- **聚合视图**："**待办**"独立 Tab，跨便签聚合所有未完成待办，提供：
  - 今日 / 未来 / 已过期 视图
  - 点击待办跳转到所在便签
- **到期与提醒**（与 3.7 联动）：
  - 单个待办项可设置到期时间与提醒时间 **[MVP]**
  - 优先级（P0/P1/P2/P3）**[V2]**

### 3.4 分组管理 **[MVP]**

- **默认分组**：
  - "**全部便签**"（不可删）
  - "**未分组**"（不可删）
  - "**回收站**"（不可删，见 3.6）
- **自定义分组**：
  - 创建、重命名、删除分组（删除分组时便签转入"未分组"）
  - 分组排序（拖拽）**[V2]**
- **分组视图**：左侧/抽屉列出所有分组，点击切换；显示每组便签数量。
- **二级分组**：暂不支持（避免过度复杂） **[迭代]**

### 3.5 搜索 **[MVP]**

- **全局搜索**：按标题 + 正文纯文本搜索
- **搜索范围**：当前分组 / 全部
- **搜索高亮**：结果中高亮命中关键词
- **搜索历史**：保留最近搜索词 **[V2]**
- **高级筛选** **[V2]**：按时间、按颜色、含/不含待办、是否过期等

### 3.6 回收站 **[MVP]**

- 删除的便签进入回收站（**不直接物理删除**）
- 回收站保留 **30 天**，超期自动清理
- 支持**单条还原**、**清空回收站**、**手动彻底删除**

### 3.7 提醒与推送 **[MVP]**

复用并简化原 Todo 文档中的推送方案。

- **触发对象**：便签内的待办项（设置了提醒时间）。
- **推送通道**：
  - **Android（PWA）**：通过 **FCM** 或 **Web Push** 在系统通知栏推送
  - **Web 端**：通过 **Web Push API** 弹出桌面通知
- **推送内容**：待办标题、所属便签标题、到期时间
- **快捷操作**：通知上支持「**标记完成**」「**稍后提醒（5/30 分钟）**」 **[V2]**
- **多次提醒**：单个待办支持多个提醒时间点 **[V2]**

### 3.8 多端同步 **[MVP]**

- **同步范围**：便签内容、分组、置顶状态、颜色、待办状态、提醒设置
- **离线优先**：
  - 所有写操作先落盘到 IndexedDB
  - 加入同步队列，待网络恢复后批量上传
- **实时同步**：通过 **SSE** 监听服务端变更，下推到其他在线设备
- **冲突策略**：
  - **MVP**：采用「最后写入优先（Last-Write-Wins）」+ `syncVersion` 字段
  - **迭代**：富文本内容考虑引入 CRDT（如 Yjs）做合并 **[迭代]**

### 3.9 个性化与外观 **[MVP / V2]**

- **便签颜色**：每条便签可标记颜色（黄/粉/蓝/绿/紫/灰，参考 WPS 便签）**[MVP]**
- **应用主题**：浅色 / 深色 / 跟随系统 **[MVP]**
- **皮肤主题**：多套配色（如莫兰迪、马卡龙等）**[V2]**
- **字号设置**：默认字号可调 **[V2]**

### 3.10 应用形态

- **PC 网页端**：
  - 浏览器访问，主要的管理与规划场景
  - 三栏布局：分组 / 便签列表 / 编辑器
  - 支持快捷键
  - **桌面悬浮便签（独立小窗）**：浏览器场景下，可通过 [Document Picture-in-Picture API](https://developer.chrome.com/docs/web-platform/document-picture-in-picture) 实现"伪悬浮"小窗 **[V2 探索]**
- **Android 端（PWA）**：
  - 安装到桌面，独立图标 + 启动画面
  - 适配移动端单栏布局
  - 推送、离线、相机/相册取图均通过 PWA 能力支持
  - **桌面小组件（Widget）**：原生 Widget 不在 PWA 能力范围，**不在本期范围**
- **不做的事**：iOS、原生 Windows/Mac 客户端、桌面便签悬浮窗（原生层面）

### 3.11 账户与隐私 **[MVP]**

- **登录方式**：**GitHub OAuth + 邮箱/密码**，二者并行支持，同账户体系
- **数据隔离**：所有数据按用户隔离
- **数据导出**：支持一键导出全部便签为 Markdown / JSON **[V2]**
- **数据备份**：自动备份至 GitHub Gist 或用户指定云存储 **[迭代]**

## 4. 个性化智能迭代规划（V2 之后）

> 这是本产品相对 WPS 便签的**核心差异化**方向，按需按兴趣排期。

1. **智能分类**：新建便签时，根据内容自动建议分组与标签。
2. **智能待办识别**：在自由书写的便签中，自动识别"明天下午 3 点开会"这类语句并建议生成待办与提醒。
3. **智能排序**："今日待办"按截止时间、优先级、个人完成习惯综合排序。
4. **智能摘要**：长便签一键生成 TL;DR。
5. **AI 续写 / 润色 / 翻译**：选中文本调用 LLM 处理。
6. **自然语言搜索**："上周写的那个关于 Next.js 的便签"。
7. **每日/每周回顾**：自动汇总本周新建、完成、未完成。
8. **代码片段优化**：识别代码块并支持语法高亮、复制按钮（针对开发者场景）。

| 维度 | 要求 |
| --- | --- |

| **安全** | HTTPS；密码 bcrypt；接口鉴权 |

## 6. 技术架构决策

### 6.1 整体架构

- **全栈框架**：**Next.js 16.2+**（App Router，LTS 至 2027-10）
  - 运行时：Node.js 20.9+，TypeScript 5.1+
  - 默认打包器：Turbopack（dev 启动 / HMR 显著更快）
  - 中间件统一使用 **`proxy.ts`** 命名（16 起 `middleware` 已弃用）
  - 所有请求 API（`cookies()` / `headers()` / `params` / `searchParams`）强制 `await`
- **前端 React**：随 Next.js 16 走 React 19.2
- **部署平台**：Vercel（自动部署 + 全球 CDN，免费额度足够自用）
- **后端基础设施**：Supabase（一体化方案，详见 6.3）

### 6.2 前端技术栈

| 类别 | 选型 |
| --- | --- |
| 语言 | TypeScript |
| UI 框架 | React 19 + Tailwind CSS + shadcn/ui |
| 状态管理 | Zustand |
| **富文本编辑器** | **Tiptap**（基于 ProseMirror，扩展性强，原生支持 task list、图片、Markdown 快捷键） |
| 服务端状态 | TanStack Query |
| 本地存储 | localForage（IndexedDB） |
| 实时通信 | **Supabase Realtime**（替代自建 SSE） |
| PWA | next-pwa + Service Worker + manifest.json |
| 表单 | React Hook Form + Zod |
| 拖拽 | dnd-kit（用于待办排序、分组排序） |

### 6.3 后端技术栈（Supabase + Next.js API Routes）

| 类别 | 选型 |
| --- | --- |
| **后端一体化** | **Supabase**（PostgreSQL + Auth + Storage + Realtime + Edge Functions 全包） |
| ORM | Prisma（连 Supabase Postgres） |
| 主数据库 | Supabase Postgres（免费层 500MB 自用足够） |
| 对象存储（图片） | **Supabase Storage** |
| 鉴权 | **Supabase Auth**（同时启用 GitHub OAuth 与 Email/Password Provider） |
| 推送服务 | FCM（Android）+ Web Push API（浏览器） |
| 任务调度 | Vercel Cron Jobs 或 Supabase pg_cron（每分钟扫描到期待办） |
| 业务 API | Next.js API Routes / Server Actions（处理推送、复杂业务） |

> **架构原则**：能用 Supabase 直连（带 RLS）的简单 CRUD 走 Supabase Client；涉及推送、批处理、隐私敏感的逻辑走 Next.js API Routes。

### 6.4 关键依赖说明

- **Tiptap**：选它而非 Lexical 的原因——文档完善、社区活跃、开箱即用的扩展（task list、image、placeholder、typography）足以覆盖 WPS 便签的编辑能力。
- **localForage**：抽象 IndexedDB API，便于离线优先实现。
- **dnd-kit**：现代化、支持触屏，适合 Android Web。

## 7. 核心模块设计要点

### 7.1 数据模型（草案）

```text
User
  id, email, passwordHash, createdAt

Group (分组)
  id, userId, name, sortOrder, createdAt
  特殊分组：UNGROUPED, TRASH 用 enum 表达，不入库

Note (便签)
  id, userId, groupId
  title, contentJson (Tiptap JSON), contentText (用于全文搜索)
  color, isPinned, isDeleted, deletedAt
  createdAt, updatedAt, syncVersion

TodoItem (从便签 contentJson 中抽取的待办，用于聚合视图与提醒)
  id, noteId, userId
  text, isDone, dueAt, remindAt
  blockId (回链到 contentJson 中的节点)
  syncVersion

PushSubscription
  id, userId, deviceType (android | web), endpoint, keys, createdAt
```

### 7.2 同步流程

1. 客户端写操作 → IndexedDB → 同步队列
2. 网络可用时上传变更，服务端基于 `syncVersion` 处理冲突
3. 服务端通过 SSE 广播变更到该用户其他在线设备
4. 客户端收到推送后合并并刷新 UI

### 7.3 推送流程

1. Vercel Cron 每分钟扫描"未来 1 分钟内 `remindAt`"的 TodoItem
2. 关联 `PushSubscription`，分别走 FCM / Web Push
3. Service Worker 接收并展示系统通知
4. 通知点击 → 打开应用并定位到对应便签

### 7.4 富文本与待办的双向同步

- 编辑器中 checkbox 状态变更 → 同步更新 `TodoItem.isDone`
- 聚合视图勾选完成 → 反向更新 `Note.contentJson` 中对应 block
